# Contributing

感谢你愿意改进 Codex Academic Paper Skills。本仓库维护的是面向真实软件工程项目论文场景的 Codex skills，因此贡献优先关注可验证、可复用、不会诱导凭空生成论文内容的改进。

## 适合贡献的内容

- 修正文档、安装步骤、示例 prompt 或术语说明中的不清楚之处。
- 补充真实项目论文规划、改稿、截图恢复、查重/AIGC 报告处理等场景案例。
- 改进 skill 工作流，使其更稳定地保留代码、SQL、配置、旧图表和手改稿证据。
- 增加轻量校验脚本或测试样例，用于检查 skill 目录结构、引用文件和元数据完整性。

## 不适合的内容

- 鼓励伪造项目、实验数据、参考文献、截图或学校材料的 prompt。
- 与软件工程论文流程无关的大型功能。
- 只为了未来可能使用而加入的复杂框架、服务或自动化。

## 开发原则

- KISS: 保持 skill 指令清晰，优先使用可读的 Markdown 和少量脚本。
- YAGNI: 只加入当前维护和用户场景已经需要的内容。
- DRY: 重复出现的规则应沉淀到共享参考文档或脚本，而不是复制到多个 prompt。
- SRP: 每个 skill 聚焦一个阶段。`academic-paper-strategist` 负责规划和证据映射，`academic-paper-composer` 负责在 working draft 上继续改稿和收口。

## 提交流程

1. Fork 本仓库并创建一个聚焦的分支。
2. 按最小范围修改文档、skill 或脚本。
3. 本地检查 Markdown 链接、目录结构和示例命令是否仍然可用。
4. 提交 PR 时说明：
   - 解决了什么问题。
   - 影响哪个 skill 或文档区域。
   - 是否改变了现有工作流或输出约束。

## Issue 建议

创建 issue 时请尽量包含：

- 使用的 Codex/Codex CLI 环境。
- 触发问题的 skill 名称。
- 期望行为和实际行为。
- 可公开的最小示例、日志片段或截图。

请不要在 issue 中上传包含个人隐私、学校账号、未公开论文全文、商业项目源码或 API key 的材料。
